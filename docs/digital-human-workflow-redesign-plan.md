# 数字人视频工厂流程重构方案

版本：V1  
日期：2026-05-23  
目标：把当前平铺式任务流程重构为可回看、可审核、可版本化、可预览的节点式生产工作台。

## 1. 结论

当前任务流程的问题不是单个按钮或样式问题，而是信息架构混在一起了：链接解析、原始输入、口播文案、口播音频、视频合成、发布选择都被铺在同一个页面中，用户无法判断当前处于哪一步，也无法稳定回看每一步产物。

新的设计应拆成两条能力线：

1. 链接解析：独立工具，不属于任务生产流程。它负责把外部链接或文本提取为可用文本，并支持一键推送到任务的原始输入。
2. 任务流程：固定五步节点，从输入到发布。无论手动模式还是全自动模式，都必须展示完整步骤和当前进度。

任务流程固定为：

输入 -> 生成口播文案 -> 生成口播音频 -> 视频合成 -> 发布

核心原则：

- 每一步都必须展示该步骤的输入、输出和历史版本。
- 手动模式下不自动跳转到下一步，用户执行完当前步骤后仍停留在当前步骤。
- 用户可以点击回到已完成步骤查看输出，但不能点击进入尚未解锁的未来步骤。
- 口播文案、口播音频、视频合成都采用版本化产物。重新生成不覆盖旧结果，而是生成 V1、V2、V3。
- 下游步骤必须明确选择上游版本，例如生成视频时选择哪一个口播音频版本，发布时选择哪一个视频版本。

## 2. 新的信息架构

### 2.1 链接解析独立出来

链接解析不再混入任务步骤，也不再作为任务中心第一步的一部分。它应该是任务工作台上方或侧边的独立工具。

链接解析流程：

输入链接/文本 -> 识别类型 -> 提取文本或下载视频 -> ASR 转写 -> 生成解析结果 -> 一键推送到原始输入

链接解析工具应包含：

- 链接输入框：支持抖音等分享文本，自动识别其中真实 URL。
- 提取按钮：点击后开始解析。
- 步骤进度：用箭头串联当前解析进度。
- 解析结果预览：展示提取文本、ASR 文本、来源标题、视频信息。
- 一键推送：把解析文本推送到当前任务的“原始输入”中。
- 推送策略：支持“覆盖当前输入”和“追加到当前输入”，默认追加。

链接解析不应该：

- 直接创建完整任务。
- 自动进入任务流程下一步。
- 依赖模型广场服务常驻。
- 把解析中间状态混入任务步骤状态。

### 2.2 任务流程固定为五个节点

任务流程只保留五个生产节点：

1. 输入
   - 输入原始文本。
   - 输入生成要求。
   - 可接收链接解析工具推送过来的文本。
2. 生成口播文案
   - 基于输入文本和生成要求生成口播文案。
   - 支持用户编辑文案。
   - 支持重新生成并保存为新版本。
3. 生成口播音频
   - 基于某一个口播文案版本生成音频。
   - 支持选择音色。
   - 支持重新生成并保存为新版本。
4. 视频合成
   - 基于数字人素材和某一个口播音频版本生成视频。
   - 支持修改视频参数。
   - 每次生成保存为新视频版本。
5. 发布
   - 选择一个视频版本。
   - 选择发布渠道。
   - 生成发布包或发布记录。

发布渠道、保护设置、本机资源等信息不应在前置步骤展示。只有到发布步骤，才展示抖音、小红书等渠道信息。

## 3. 模式定义

### 3.1 手动模式

把“审核模式”改名为“手动模式”。

手动模式的行为：

- 全部五个节点始终可见。
- 当前可执行节点高亮。
- 已完成节点可点击回看。
- 当前节点可点击。
- 未来节点不可点击，展示锁定态。
- 执行完当前节点后，不自动跳转下一节点。
- 用户确认后，才手动进入下一节点。
- 文案、音频、视频都可以多次生成，每次生成新版本。

手动模式适合调参、人工审稿、选择版本、商业化内容生产。

### 3.2 全自动模式

全自动模式也必须展示完整流程节点。

全自动模式的行为：

- 全部五个节点始终可见。
- 系统自动按节点顺序执行。
- 当前运行节点高亮，并展示运行日志和当前输出。
- 用户可以点击已完成节点查看历史输出。
- 未来节点仍不可点击。
- 执行完成后停在最后完成节点或展示完成态，不把所有信息一次性平铺。

全自动模式适合快速生成，但不能牺牲过程可见性。

## 4. 节点式 UI 方案

### 4.1 顶部流程节点

桌面端建议横向节点：

输入 -> 生成口播文案 -> 生成口播音频 -> 视频合成 -> 发布

移动端建议纵向节点：

输入
  ↓
生成口播文案
  ↓
生成口播音频
  ↓
视频合成
  ↓
发布

每个节点展示：

- 节点名称。
- 状态：未开始、可执行、运行中、已完成、失败、锁定。
- 最近一次输出摘要。
- 版本数量，例如 V1/V2。
- 错误提示入口。

### 4.2 点击规则

假设当前执行到第 3 步“生成口播音频”：

- 可以点击第 1 步“输入”。
- 可以点击第 2 步“生成口播文案”。
- 可以点击第 3 步“生成口播音频”。
- 不能点击第 4 步“视频合成”。
- 不能点击第 5 步“发布”。

解锁规则：

- 第 2 步需要第 1 步已保存输入。
- 第 3 步需要至少一个口播文案版本。
- 第 4 步需要至少一个口播音频版本和一个数字人素材。
- 第 5 步需要至少一个视频版本。

### 4.3 每一步的页面结构

每个步骤页面统一分为三块：

1. 当前输入
   - 展示本步骤依赖的上游内容。
   - 如果有多个上游版本，必须提供版本选择器。
2. 执行区
   - 展示当前步骤可编辑参数。
   - 展示执行按钮，例如“生成 V2”。
   - 长任务必须有 loading、进度、日志。
3. 输出区
   - 展示该步骤所有版本。
   - 当前选中版本可预览。
   - 可设置某个版本为下游默认使用版本。

## 5. 版本模型

### 5.1 版本命名

版本名称不允许用户自定义，统一使用：

- V1
- V2
- V3

版本标题可以由系统辅助生成，但版本号必须稳定、简单。

### 5.2 口播文案版本

每次生成或修改保存后，形成一个新的口播文案版本。

字段建议：

- id
- projectId
- versionNo
- label，例如 V1
- sourceInputSnapshot
- requirementsSnapshot
- scriptText
- title
- modelInfo
- createdAt
- status

输出展示：

- 文案全文。
- 标题和摘要。
- 生成模型。
- 创建时间。
- 使用此版本生成音频按钮。

### 5.3 口播音频版本

每次合成口播音频后，形成一个新的音频版本。

字段建议：

- id
- projectId
- versionNo
- label，例如 V1
- sourceScriptVersionId
- voiceId
- voiceName
- audioUri
- duration
- transcriptText
- modelInfo
- createdAt
- status

输出展示：

- 音频播放器。
- 对应文案版本。
- 音色信息。
- 时长。
- 使用此版本合成视频按钮。

### 5.4 视频版本

每次视频合成后，形成一个新视频版本。

字段建议：

- id
- projectId
- versionNo
- label，例如 V1
- sourceAudioVersionId
- sourceScriptVersionId
- avatarAssetId
- videoSettings
- videoUri
- duration
- resolution
- qualityReport
- createdAt
- status

输出展示：

- 视频播放器。
- 对应音频版本。
- 数字人素材。
- 视频参数。
- 分辨率和时长。
- 发布此版本按钮。

### 5.5 发布版本

发布步骤必须选择一个视频版本。

发布记录字段建议：

- id
- projectId
- videoVersionId
- channels
- publishPackageUri
- copyText
- status
- createdAt

## 6. 下游选择规则

下游步骤不能默认偷用“最后一次生成”的产物，必须让用户可见地选择。

默认选择策略：

- 如果用户已经手动选择过上游版本，沿用用户选择。
- 如果没有选择，默认选择最新的成功版本。
- 如果只有一个版本，自动选中但仍展示选择器。

典型场景：

- 用户生成了口播文案 V1、V2。
- 在“生成口播音频”步骤，必须选择基于 V1 还是 V2。
- 用户生成了音频 V1、V2、V3。
- 在“视频合成”步骤，必须选择使用哪一个音频版本。
- 用户生成了视频 V1、V2。
- 在“发布”步骤，必须选择发布 V1 还是 V2。

## 7. 任务页面建议布局

### 7.1 页面顶部

顶部展示：

- 任务标题。
- 模式切换：手动模式 / 全自动模式。
- 任务状态。
- 当前节点流程条。

任务标题规则：

- 用户可编辑标题。
- 用户不填时，由模型基于原始输入自动生成标题。

### 7.2 左侧或上方：链接解析工具

链接解析工具应作为独立卡片或抽屉，不占用任务流程主体。

建议布局：

输入框 + 提取按钮  
输入链接/文本 -> 识别类型 -> 提取/下载 -> ASR -> 解析结果  
解析结果预览 + 推送到原始输入

### 7.3 主体：步骤工作区

主体只展示当前选中步骤的内容，不把所有步骤内容平铺出来。

每一步使用统一结构：

- 上游输入
- 参数和操作
- 输出版本

### 7.4 右侧：版本和执行记录

右侧可以放：

- 当前步骤版本列表。
- 最近执行日志。
- 错误信息。
- 对比入口。

移动端改为底部抽屉或折叠区。

## 8. API 设计建议

### 8.1 链接解析

POST /api/source-extractions

输入：

- sourceText

输出：

- extractionId
- detectedType
- sourceUrl
- title
- extractedText
- transcriptText
- mediaUri
- status

POST /api/projects/:id/input/apply-source

输入：

- extractionId
- mode: append | replace

输出：

- updatedInputText

### 8.2 口播文案版本

POST /api/projects/:id/script-versions

输入：

- inputText
- requirements
- modelId

输出：

- scriptVersion

PATCH /api/projects/:id/script-versions/:versionId

用途：

- 用户编辑文案后保存为新版本。

### 8.3 音频版本

POST /api/projects/:id/audio-versions

输入：

- scriptVersionId
- voiceId
- ttsModelId

输出：

- audioVersion

### 8.4 视频版本

POST /api/projects/:id/video-versions

输入：

- audioVersionId
- avatarAssetId
- videoSettings

输出：

- videoVersion

### 8.5 发布

POST /api/projects/:id/publish-records

输入：

- videoVersionId
- channels

输出：

- publishRecord

## 9. 数据迁移建议

当前项目里已有 artifacts.script、artifacts.audio、artifacts.video 和 versions。建议迁移为显式版本集合：

- scriptVersions
- audioVersions
- videoVersions
- publishRecords

兼容策略：

- 老的 artifacts.script 迁移为 scriptVersions.V1。
- 老的 artifacts.audio 迁移为 audioVersions.V1。
- 老的 artifacts.video 迁移为 videoVersions.V1。
- 老的 project.versions 可以合并到 videoVersions。

迁移时不要删除旧文件，只补元数据索引。

## 10. P0 实施范围

P0 只做流程主干，不做高级对比。

P0 必须完成：

- 链接解析从任务流程中抽离。
- 增加链接解析独立步骤条。
- 支持一键推送解析文本到原始输入。
- 审核模式改名为手动模式。
- 全自动模式和手动模式都展示完整流程节点。
- 节点点击规则：可回看已完成，不可越级去未来步骤。
- 任务主体改为“当前步骤工作区”，不再平铺全部信息。
- 文案、音频、视频版本化。
- 音频和视频版本可实时预览。
- 视频发布时选择视频版本。
- 生成文案、生成音频、生成视频都必须有 loading、toast、错误反馈和执行日志。

P0 暂不做：

- 多版本并排 A/B 对比。
- 视频质量自动评分。
- 云端多人协作。
- 自动发布到平台。

## 11. P1 增强方向

P1 可以做：

- 版本并排对比：文案对比、音频试听对比、视频并排播放。
- 分支关系可视化：例如视频 V3 来自音频 V2，音频 V2 来自文案 V1。
- 版本备注：允许用户对版本写内部备注，但版本名仍为 V1/V2。
- 任务模板：常用口播类型、视频参数、音色和发布格式。
- 自动质检：检测口型、清晰度、音频响度、视频分辨率。
- 发布包管理：封面、标题、正文、标签、视频版本统一打包。

## 12. 验收标准

验收时必须逐条检查：

- 链接解析不出现在任务五步流程里。
- 链接解析有独立进度：输入链接/文本 -> 识别类型 -> 提取/下载 -> ASR -> 解析结果。
- 解析结果可以一键推送到原始输入。
- 任务流程固定展示：输入 -> 生成口播文案 -> 生成口播音频 -> 视频合成 -> 发布。
- 手动模式名称已替换审核模式。
- 手动模式执行完某一步后不自动跳到下一步。
- 全自动模式也展示同样的节点流程。
- 当前执行到第三步时，可以点击第一、第二、第三步，不能点击第四、第五步。
- 每一步都能看到对应输出内容。
- 重新生成文案会产生 V2，不覆盖 V1。
- 重新生成音频会产生 V2，不覆盖 V1。
- 重新生成视频会产生 V2，不覆盖 V1。
- 生成音频时可以选择文案版本。
- 合成视频时可以选择音频版本。
- 发布时可以选择视频版本。
- 音频版本可以直接试听。
- 视频版本可以直接预览。
- 发布渠道只在发布步骤出现。

## 13. 关键产品判断

这个重构的核心不是“多放几个按钮”，而是把任务从线性黑盒改成可审阅的生产流水线。

对商业化产品来说，用户最关心的是：

- 当前做到哪一步。
- 每一步生成了什么。
- 我能不能回看和修改。
- 修改后旧版本会不会丢。
- 下一个步骤到底用了哪个版本。
- 最终发布的是哪个视频。

如果这些问题都能在界面上一眼看清，整个产品的可信度会明显提升。
